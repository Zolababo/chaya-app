# 점주 계정 정책 (파일럿)

**결정:** 2026-06-01  
**우선순위 문서:** `docs/CHAYA_PRODUCT_PRIORITY_PILOT.md`

---

## 한 줄

**매장(테넌트)당 로그인 계정 1개**를 쓰고, **여러 기기에서 동시 로그인**한다.  
파일럿 단계에서는 **직원별 계정·앱 내 직원 초대·역할 분리 UI**를 만들지 않는다.

---

## 의도

| 하지 않음 (당분간) | 이유 |
|-------------------|------|
| Owner 앱 내 **직원 초대** | RLS·역할·승인·삭제·초대 메일·감사(멤버 관리) 부담 |
| 점주앱을 **소장 / 직원 앱**으로 나눔 | 아직 매장 네트워크·운영 모델이 단순함 |
| mailto로 **직원 추가 요청** | 위와 동일 — 정책상 1계정·다기기로 대체 |

| 대신 함 | 예 |
|--------|-----|
| **동일 계정**으로 여러 기기 로그인 | 사장 폰 · 매장 태블릿 · 주방 태블릿 |
| Supabase Auth **세션은 기기마다** 유지 | 같은 이메일·비밀번호로 `/m/login` 반복 |
| **P0** 비밀번호 찾기·변경 | 계정 1개이므로 소장이 스스로 복구 |

Supabase는 기본적으로 **동일 사용자의 여러 세션(기기)을 허용**한다. 별도 “기기 등록” 기능 없이도 동시 사용이 가능하다.

---

## Ops·DB와의 관계

- **신규 매장·소장 첫 계정:** 계속 **`/ops/merchants`** 초대·승인 (`docs/MERCHANT_FIELD_ONBOARDING.md`).
- DB **`merchant_tenant_members`**·역할(`staff`, `menu_editor` 등)은 **코드/정책용으로 유지**할 수 있으나, 파일럿 UX는 **owner 1행 + 1 Auth 사용자**를 전제로 한다.
- 테스트·예외로 Ops가 **추가 멤버**를 넣은 경우만 `/m/.../more/staff` 에 조회용으로 보일 수 있다 — **앱에서 추가·초대하지 않음**.

---

## 점주 UI

- **더보기 → 로그인 · 기기** (`/m/{tenant}/more/staff`) — 다기기 안내, 비밀번호·계정 링크.
- **직원 초대 mailto** 제거.
