# 점주·소비자 앱 상생 가이드 (Single source of truth)

**목표:** 같은 테넌트·같은 주문 파이프라인 위에서 `/t/*` 와 `/m/*` 가 **어긋나지 않고** 운영·개발·배포 판단을 같이 내릴 수 있게 한다.  
형식은 고정하지 않으며, 상성이 깨지는 지점이 보이면 **이 문서를 먼저** 고친다.

---

## 1. 공유 식별자 (반드시 일치)

| 개념 | 손님 `/t` | 점주 `/m` | 비고 |
|------|-----------|-----------|------|
| 가게 | `tenant_slug` (URL·DB) | 동일 | `ChayaMenus.tenant_slug`, `orders.tenant_slug` |
| 주문 | `orders.id` (UUID) | 동일 | 손님 상세·점주 큐가 같은 행을 본다 |
| 게스트 세션 | `guest_session_id` + 쿠키/저장소 | (직접 쓰지 않음) | RPC·RLS는 `BARRIER_FREE_NEXT_STEPS.md` 순서 |

**상성 깨짐 신호:** 한쪽만 마이그레이션 적용, `tenant_slug` 표기 불일치, 게스트 쿠키 없이 주문 상세만 북마크 등.

---

## 2. 주문 생애주기 (누가 무엇을 보나)

| 단계 | 손님 앱 | 점주 앱 | 서버·알림 (요약) |
|------|---------|---------|-------------------|
| 메뉴 탐색 | `/t/{tenant}` | (참고 링크만) | 동일 `ChayaMenus` |
| 장바구니·제출 | `/t/.../cart` | — | 게스트 액션·주문 INSERT |
| 신규 주문 인지 | `/t/.../orders` 허브·상세 | `/m/.../orders`, 대시보드 | `merchant_notification_events`, 선택 Resend·푸시·웹훅 |
| 상태 변경 | 조회(폴링/표시) | `owner`/`staff` 만 변경 | service role + 감사 로그 |
| 메뉴·품절 | 반영된 메뉴만 노출 | `owner`/`menu_editor` | 손님 화면은 캐시·새로고침 정책과 맞물림 |

**상성 원칙:** 손님이 보는 상태는 **점주가 바꾼 값의 최종본**이어야 한다. 손님 쪽에서 “가짜 진행률”을 두지 않는다.

---

## 3. URL·권한 경계 (역할 혼동 방지)

| 경로 패턴 | 사용자 | 인증 |
|-----------|--------|------|
| `/t/{tenant}/*` | 손님 | 게스트 세션 + anon RPC (RLS) |
| `/m/*` | 점주·직원 등 | Supabase Auth + `merchant_tenant_members` |
| `/ops/*` | 플랫폼 | 별도 운영자 테이블 |

**상성:** 손님 경로에 **점주 비밀**을 싣지 않는다. 점주 주문 처리에 **손님 anon만**으로는 부족한지는 RLS·서버 가드로 이미 분리됨 — 변경 시 이 표를 갱신한다.

---

## 4. 운영·환경 (양쪽 다 걸리는 것)

| 항목 | 손님 | 점주 | 메모 |
|------|------|------|------|
| `NEXT_PUBLIC_SUPABASE_*` | 필요 | 세션 필요 | 동일 프로젝트 전제 |
| `SUPABASE_SERVICE_ROLE_KEY` | 서버 주문·(일부) 조회 | 주문·메뉴·감사 | Vercel 서버만 |
| `NEXT_PUBLIC_SITE_URL` | 메일 링크 품질 | 메일 링크 | 없으면 `VERCEL_URL` |
| Resend / VAPID / 웹훅 | 손님 직접 X | 점주 알림 | 파이프라인은 `merchant-notification-*` |

`GET /health` 는 **배포 단위**에서 두 앱이 같은 설정을 쓰는지 빠르게 보는 용도(비밀 미노출).

## 5. 품질 게이트 (둘을 같이 본다)

| 도구 | 의미 |
|------|------|
| `pnpm smoke:consumer` | `/t` 단독 |
| `pnpm smoke:merchant` (있다면) | `/m` 단독 |
| `pnpm smoke:platform` | **소비자→점주 연속** 시나리오 (문서: `DEPLOY_VERCEL.md`) |
| `MERCHANT_PARALLEL_VALIDATION_*` | 병행·컷오버 시 양쪽 동시 |

**상성:** PR이 `/t` 만 건드려도, **주문이 점주 큐에 보이는지** 한 줄이라도 검증한다.

---

## 6. 알려진 긴장점 (의도적으로 문서화)

- **게스트 콜드 링크:** 주문 상세 URL만 처음 열면 세션 쿠키가 없을 수 있음 → 앱의 재시도·안내와 점주 쪽 “주문 ID 공유” 교육이 맞물려야 함 (`QR_AND_GUEST_ORDERS.md`).
- **서버리스 쿨다운:** 알림은 인스턴스 메모리 한계 — 손님 체감과 점주 체감이 다를 수 있음.
- **역할 세분:** `menu_editor` 는 주문 상태를 못 바꿈 — 손님 화면은 “왜 안 바뀌지”가 아니라 점주 내부 역할 문제로 남도록 UX 문구 정리.

## 7. 다른 문서와의 관계 (전면 수정 가능)

| 문서 | 역할 |
|------|------|
| `MERCHANT_OPERATIONS_ROADMAP.md` | 점주 Phase·진척% (점주 깊이) |
| `BARRIER_FREE_NEXT_STEPS.md` | 손님 RPC·접근성·SQL 순서 (손님 깊이) |
| **본 문서** | **두 앱이 만나는 지점만** — 중복 서술은 줄이고 링크로 대체 가능 |

진척 % 는 **영역별로 따로** 두되, “한 가게 스토리”를 말할 때는 **본 문서 + 병행 체크리스트**를 우선한다.
