# Merchant 실사용 실행 계획 (1차)

이 문서는 점주앱을 "테스트 가능" 상태에서 "실사용 가능" 상태로 올리는 최소 실행안입니다.

## 0) 목표

- 점주가 `m/login` 후 근무 시간 동안 재로그인 없이 운영 가능
- 주문 상태 변경/메뉴 수정이 실제 매장에서 반복 가능
- 운영자가 점주 초대/권한/tenant 연결을 표준 절차로 처리 가능

## 1) 사전 조건 (완료 여부)

- [ ] `/m/login` 이메일 로그인 정상
- [ ] `/ops/merchants` 점주 초대 정상
- [ ] `merchant_tenant_members` 연결 확인
- [ ] 자동 로그아웃 이슈 재현되지 않음 (최소 20분 유지 테스트)

## 2) 실행 순서 (권장 1~2일)

### A. 운영자 준비

- [ ] 운영자 계정 1명 이상 `platform_operators` 등록
- [ ] 실사용 대상 tenant slug 확정 (예: `demo`, `gangnam-1ho`)
- [ ] owner/staff 권한 정책 확정 (누가 메뉴 수정 가능한지)

### B. 점주 계정/권한 준비

- [ ] 각 tenant에 owner 계정 최소 1개 배정
- [ ] staff 계정(필요 시) 배정
- [ ] 점주별 로그인 테스트 (실제 모바일 포함)

### C. 메뉴/카테고리 데이터 준비

- [ ] 대표 메뉴 10개 이상 등록
- [ ] 카테고리/정렬 순서 확인
- [ ] 품절 토글 동작 확인
- [ ] 소비자 `/t/{tenant}` 반영 확인

### D. 주문 운영 리허설

- [ ] `/m/{tenant}/dashboard` 에서 최근 24시간 접수·대기 요약 확인
- [ ] 손님 주문 생성 -> 점주 주문 큐 반영
- [ ] 상태 전이: `pending -> accepted -> preparing -> ready -> completed`
- [ ] 점주 재접속/탭 전환 후 상태 유지 확인

## 3) SQL 점검 루틴

실행 파일:

- `supabase/scripts/verify_merchant_live_readiness.sql`

실행 타이밍:

- 오픈 전 1회
- 첫 실사용 당일 마감 후 1회

통과 기준:

- owner 존재
- 메뉴 개수 0 아님
- 최근 주문/상태 통계 정상 집계

## 4) 장애 시 즉시 대응

- 로그인 불가: `/m/login` -> 이메일/비번 재확인 -> Supabase Users 상태 확인
- 접근 거부: `merchant_tenant_members`에 user_id/tenant_slug 연결 여부 확인
- 주문 상태 저장 실패: service role/env/DB 정책 확인 후 재시도

## 5) 컷오버 판단 체크

- [ ] 최근 2일 동안 주문 누락/중복 없음
- [ ] 점주 1명 이상이 모바일 단독으로 운영 완료
- [ ] 운영자가 점주 초대/권한 변경을 문서대로 재현 가능
- [ ] fallback(기존 운영 경로) 연락 체계 준비 완료
