# QR 접근·테넌트 식별 & 비회원 주문 설계 (초안)

메뉴 앱은 식당 QR로 진입하고, **모바일 퍼스트·반응형**으로 동일 URL이 태블릿·PC에서도 동작한다는 전제다.

---

## 1. QR에 넣을 URL (테넌트 식별)

### 1.1 권장 형태: 경로에 테넌트 키

**예시**

- `https://menu.example.com/t/{tenant_slug}`  
- 또는 짧게: `https://menu.example.com/r/{short_code}`  

**`tenant_slug`**

- 사람이 읽기 쉬운 가게 식별자(영문·숫자·하이픈). QR 인쇄물·홍보에 재사용하기 좋다.
- DB의 `tenants` 테이블에 **유니크**로 둔다.

**메뉴 테이블(`ChayaMenus`)과의 연결 (현재 구현)**

- 소비자 앱은 `/t/{tenant_slug}` 의 `{tenant_slug}` 와 동일한 값이 **`ChayaMenus.tenant_slug`** 에 들어 있는 행만 조회한다.
- 마이그레이션: `supabase/migrations/20260415130000_chaya_menus_tenant_slug.sql` 를 적용하면 기존 행은 기본값 `demo` 로 채워진다. 새 매장은 행마다 해당 가게 slug 로 `tenant_slug` 를 맞춘다.
- 로컬에 Supabase env 가 없으면 앱은 기존처럼 **데모 목록**으로 동작한다 (`NEXT_PUBLIC_SUPABASE_*`).

**`short_code`** (선택)

- QR 데이터량을 줄이려면 **6~10자 정도의 짧은 코드**만 인쇄하고, 서버/앱에서 `short_code → tenant_id`로 조회한다.
- 짧은 코드는 **충돌 방지·무차별 대입 난이도**를 고려해 충분한 엔트로피(난수)를 쓴다.

### 1.2 서브도메인 방식 (`bukchang.menu.example.com`)

- 테넌트마다 DNS/인증서를 나누면 운영 부담이 커진다.
- **와일드카드 인증서 + 와일드카드 DNS**가 없으면 초기에는 비권장. 나중에 브랜딩이 필요해지면 검토.

### 1.3 보안에 대한 짧은 주의

- URL의 `slug`/`short_code`는 **비밀이 아니다**(QR은 누구나 찍을 수 있음).
- 진짜 격리는 **RLS + `tenant_id` 검증**으로 한다. “다른 가게 데이터가 보이지 않게”가 기본이다.

### 1.4 반응형

- 동일한 라우트에서 **뷰포트에 따라** 그리드·폰트·터치 타겟만 조정한다.
- “PC 전용 URL”을 따로 두지 않아도 된다.

## 2. 비회원 주문 식별 & 나중에 계정으로 묶기

### 2.1 목표

- **주문은 비회원으로도 가능**해야 한다.
- 이후 가입/로그인하면 **같은 기기·같은 익명 세션에서 남긴 주문**을 (정책에 따라) **한 계정으로 묶을 수 있어야** 한다.

### 2.2 익명 세션 키 `guest_session_id`

- 첫 방문(또는 첫 “주문 시작”) 시 **브라우저에만** 존재하는 식별자를 둔다.
- 생성: **암호학적으로 안전한 난수**(예: 128bit 이상), `localStorage` 또는 1st-party 쿠키에 저장.
- Supabase `orders`(또는 별도 `guest_sessions`)에 **`guest_session_id`**(또는 해시 저장)를 기록해 **주문과 연결**한다.

**레포에 포함된 마이그레이션 적용 순서(파일명·RLS·RPC)**는 한 문서에 모아 두었다: [배리어프리 진행·손님 주문 RPC 적용 순서](./BARRIER_FREE_NEXT_STEPS.md) 의 「Supabase 손님 주문」절. (`apps/consumer-menu/.env.example` 주석과 동일 계열.)

**현재 `consumer-menu` 구현(요약)**

- 세션 값은 `localStorage`(`chaya_guest_session`)에 두고, 서버 RPC(`get_order_for_guest` 등)와 맞추기 위해 **동일 키 이름의 1st-party 쿠키**로도 동기화한다(`GuestSessionCookieSync`, 주문 전 `syncGuestSessionCookieFromBrowser`). 주문 상세 첫 페인트에서 쿠키가 늦을 수 있어 **`GuestOrderDetailSessionRetry`**로 세션이 있으면 한 번만 `router.refresh()` 한다.
- 주문 완료·찾을 수 없음·주문 허브·장바구니 등에 **같은 폰·같은 브라우저** 안내 문구를 넣어 두었고, 접수 화면에는 **주문 URL 복사**(클립보드)·**Web Share API**가 가능한 환경에서만 **다른 앱으로 공유** 버튼·`aria-live` 피드백이 있다. 세부·RPC 적용 순서는 [배리어프리 진행 문서](./BARRIER_FREE_NEXT_STEPS.md)를 따른다. 배포 후 DB 점검은 `supabase/scripts/verify_guest_order_rpcs.sql` 로 한 번에 확인할 수 있다.

**한계**

- 시크릿 모드·쿠키 삭제·기기 변경 시 세션은 끊긴다 → 이 경우 **“과거 주문 찾기”는 전화·주문번호·영수증 코드** 같은 **2채널 인증**으로 보완하는 식으로 단계적으로 넣을 수 있다.

### 2.3 주문 행에 두면 좋은 필드 (개념)

| 필드 (개념) | 설명 |
|-------------|------|
| `tenant_id` | 어느 가게 주문인지 |
| `guest_session_id` | 비회원 세션과의 연결 |
| `user_id` | 로그인한 경우만 채움, 비회원은 `NULL` |
| `status` | `pending` → `approved` 등 |

### 2.4 가입 후 “묶기(claim)”

1. 사용자가 **회원 가입 또는 로그인**한다.
2. 클라이언트가 **현재 `guest_session_id`**를 서버에 보낸다(HTTPS, 짧은 만료 토큰 권장).
3. **서버(Edge Function 또는 Next Route)**에서:
   - 세션 소유가 합리적인지 검증한 뒤,
   - 해당 `guest_session_id`로 남긴 주문 중 **아직 `user_id`가 비어 있는 행**에 `user_id`를 채운다.

민감도가 높으면 **“최근 N일 이내 주문만”** 같은 정책을 둔다.

### 2.5 개인정보·수집 최소화

- 비회원만으로도 주문하려면 **테이블 번호·메뉴**는 필수, **전화번호는 선택** 같은 식으로 단계를 나눈다.
- 전화번호로 나중에 병합할 계획이면, **수집 시점·목적**을 화면에 명시하는 것이 안전하다.

## 3. 다음 합의하면 좋은 것

- [ ] QR에는 **`/t/{slug}`** vs **`/r/{short_code}`** 중 무엇을 기본으로 할지.
- [ ] 비회원 주문에 **테이블 번호**를 필수로 할지(매장 운영 방식).
- [ ] “과거 주문 찾기”를 **세션 묶기만** 할지, **주문번호+전화**까지 갈지.

### 3.1 운영 제안 초안 (**미확정** · 회의 전까지 기본 가정으로만 사용)

실제 채택은 팀 결정 후 위 체크박스를 갱신한다.

| 주제 | 제안 초안 | 이유(짧게) |
|------|-----------|-----------|
| QR 기본 URL | **`/t/{tenant_slug}`** 를 1안으로 유지, `/r/{short_code}`는 **QR 데이터 절약**이 필요할 때만 2단계로 도입 | 이미 구현·문서·DB 연결이 slug 기준으로 맞춰져 있음. short_code는 DB·라우팅 추가 비용 있음. |
| 테이블 번호 | **선택** 유지(기본 필수 비권장) | 카페형·포장 등 테이블이 없을 수 있음. 필수 전환 시 앱 검증 메시지·INSERT RLS·UI를 같이 고쳐야 함. |
| 과거 주문 찾기 | **1단계**: 세션(같은 폰·같은 브라우저) + 현재 UX 안내.**2단계 옵션**: 주문번호+연락처(또는 OTP)·만료 시간 | 세션 없이 URL만 알면 타인 주문이 보일 수 있는 모델이면 추가 인증 없이 과거 조회 확장은 위험이 큼. |

## 4. 백로그 — 손님 주문·QR 남용 방지 (**KEEP · 필수 후속**)

**상태 (2026-06):** **1차 구현 완료** — 서명·만료 테이블 QR, 주문 submit 검증, rate limit, 영업시간 자동 중지.  
**지금 유지(변경 금지):** 주문 상세·목록·상태 RPC의 **`guest_session_id` 일치** — 공유된 `…/orders/{id}` URL만으로 타인·다른 브라우저에서 주문 내용이 보이지 않게 하는 장치. 이 동작은 **제거하지 않는다.**

### 4.0 구현 요약 (2026-06)

| # | 기능 | 구현 |
|---|------|------|
| 1 | 서명·만료 QR | `/t/{tenant}/qr?table=&tsig=` 스캔 → **HttpOnly 쿠키**(스캔 시점+TTL) → 메뉴 리다이렉트. `?table=`만 복사 시 주문 불가. 인쇄 QR은 **만료 없음**. |
| 2 | 주문 받기 중지 | `tenant_store_settings.orders_accepting` (기존) |
| 3 | rate limit | `submitGuestOrderAction` — IP·guest_session (15분 창) |
| 4 | 영업시간 | `business_open`/`business_close` 설정 시 KST 구간 밖 `orders_closed` |
| 5 | 의도적 주문 공유 | **미구현** (영수증·share 코드 — 파일럿 제외) |

**운영:** 마감·이상 시 점주 **주문 받기 중지** (`/m/{tenant}/more` 영업 설정). 테이블 QR **재인쇄** 시 새 토큰 발급.

### 4.1 문제 (왜 메뉴 QR도 막아야 하는지)

- **주문 URL**은 위 세션 규칙으로 **조회**가 막혀 있음.
- **메뉴 QR URL** (`/t/{slug}`, `?table=` 등)은 **그대로 공유·북마크·외부 접속**이 가능하고, **새 주문 접수**까지 이어짐 → 장난·외부 연타 주문에 취약.
- “공유 링크만 차단”은 카카오·문자 등 채널을 구분할 수 없으므로, **세션 검사만으로 메뉴 URL을 막는 방식은 불가**에 가깝다. **입장권(토큰)·운영 스위치**가 필요하다.

### 4.2 필수 후속 구현 (우선순위 제안)

| # | 기능 | 요약 |
|---|------|------|
| 1 | **서명·만료 QR(테이블) 토큰** | QR에 `token=`(또는 `/r/{code}`) — 없거나 만료 시 **주문 제출 거부**(메뉴 읽기만 허용할지는 합의). 정적 slug만으로는 외부 주문 차단 불가. |
| 2 | **점주 「주문 받기 중지」 토글** | `/m/[tenant]` 에서 즉시 손님 **주문 INSERT 차단** + 손님 앱 안내 문구. |
| 3 | **주문 제출 rate limit** | IP·`guest_session` 기준 (예: 15분 N건). 로그인용 `simple-rate-limit` 패턴 재사용. |
| 4 | (선택) **의도적 주문 공유** | 만료·1회용 `share` 코드로 **조회만** 허용 — 지금의 “세션 없으면 404”와 별도 제품 기능. |
| 5 | (선택) **영업시간·테이블 PIN** | 운영 정책에 따라 1·2와 조합. |

### 4.3 구현 시 건드릴 곳 (참고)

- DB: `tenants` 또는 별도 설정 행(주문 중지), 토큰 검증 RPC/정책, 필요 시 RLS·`orders` INSERT 조건.
- 손님: `/t/[tenant]` 진입·`submitGuestOrder`·장바구니 CTA 전 검증.
- 점주: 대시보드·설정 UI, 감사 로그(토글 변경).
- 문서: 본 §4 체크리스트 갱신, `MERCHANT_OPERATIONS_ROADMAP.md` C6 행, `.env.example` 주석.

### 4.4 하지 않는 것 (이번 보류 범위)

- 메뉴 URL에 **주문 상세와 동일한 `guest_session` 차단**을 그대로 적용 — 첫 QR 스캔과 충돌.
- 카카오·인앱 브라우저 **정책 대응만**으로 남용 방지를 대체.
